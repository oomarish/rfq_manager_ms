rfq

Core — anchor record for every Request for Quotation.
Column Type Constraint
id UUID PK
workflow_id UUID FK → workflow, NOT NULL
current_stage_id UUID FK → rfq_stage, NULL
name VARCHAR(200) NOT NULL
client VARCHAR(150) NOT NULL
industry VARCHAR(100)
country VARCHAR(80)
priority ENUM DEFAULT 'normal'
deadline DATE NOT NULL
owner VARCHAR(100) NOT NULL
description TEXT
status ENUM NOT NULL, DEFAULT 'Draft'
progress INT DEFAULT 0
outcome_reason VARCHAR(500)
created_at TIMESTAMP DEFAULT NOW
updated_at TIMESTAMP DEFAULT NOW
priority: normal critical
status: Draft In preparation Submitted Awarded Lost Cancelled
outcome_reason (not loss_reason) — covers all outcomes: why awarded, why lost, why cancelled.
Draft = saved without finalizing (no stages generated, current_stage_id = NULL, progress = 0). GET /rfqs excludes Drafts by default — pass ?status=Draft to see them. On finalization, backend generates stages and sets status = "In preparation".

rfq_stage

Tracking — a specific RFQ's instance of a stage, with time tracking and blocker state.
Column Type Constraint
id UUID PK
rfq_id UUID FK → rfq, NOT NULL
stage_template_id UUID FK → stage_template, NOT NULL
name VARCHAR(100) NOT NULL (snapshot)
order INT NOT NULL (snapshot)
assigned_team VARCHAR(100)
status ENUM DEFAULT 'Not started'
progress INT DEFAULT 0
planned_start DATE
planned_end DATE
actual_start DATE
actual_end DATE

mandatory_fields TEXT (snapshot)
blocker_status ENUM
blocker_reason_code VARCHAR(50)
created_at TIMESTAMP DEFAULT NOW
updated_at TIMESTAMP DEFAULT NOW
status: Not started In Progress Completed Skipped
blocker_status: Blocked Resolved or NULL (never blocked) — orthogonal to status
blocker_reason_code examples: supplier_delay missing_client_docs awaiting_internal_approval technical_clarification workbook_issue budget_hold pending_lab_results
status = workflow progression. blocker_status = overlay flag (a stage can be "In Progress" and "Blocked" simultaneously). Removed Blocked from status enum to avoid redundancy with blocker_status. Skipped is for No-Go scenarios: remaining stages are set to Skipped when an RFQ is cancelled.

No FK to reminder — the relationship is behavioral (batch reads rules → creates reminders), not relational.

Batch Processing Rules

Cron job logic for reminder lifecycle management.
  • Runs every 15 min (configurable)
  • Checks due_date <= today AND status != resolved
  • Updates status to overdue when late
  • Sends notifications based on type (internal / external)
  • Uses last_sent_at to rate-limit (no re-send within X hours)
  • Increments send_count on each send, logs success/failure
