Step 1

Settings

config/settings.py
Understand

What is BaseSettings? Pydantic has a special class for configuration. Instead of os.getenv("DATABASE_URL") everywhere, you define one class that reads .env automatically and validates nothing is missing. If DATABASE_URL isn't set, the app crashes at startup with a clear error — not 10 minutes later. Pydantic

Build

Settings class with BaseSettings. Fields: DATABASE_URL (required), APP_ENV (default "development"), APP_PORT (default 8000), IAM_SERVICE_URL, EVENT_BUS_URL, FILE_STORAGE_PATH, CORS_ORIGINS, JWT_SECRET. Add model_config = SettingsConfigDict(env_file=".env"). Export: settings = Settings(). Update .env.example.

Verify

Python shell: from src.config.settings import settings → print(settings.DATABASE_URL). Delete DATABASE_URL from .env → re-import → crashes. Put it back.

Step 2

Database connection

database.py
Understand

Three objects. engine = connection to PostgreSQL (once). SessionLocal = factory that creates sessions. get_db() = generator using yield — one session per request, closed when done. SQLAlchemy yield

Critical: get_db() does NOT commit(). The controller decides when. This is transaction safety — RFQ + 5 stages = one session, one commit, one rollback if anything fails.

Build

Import settings. engine = create_engine(settings.DATABASE_URL). SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine). Base = declarative_base(). get_db() with try/yield/finally.

Ask Claude

"What does autocommit=False and autoflush=False mean, and why both off?" — defense question.

Verify

Python shell: from src.database import engine → engine.connect(). PostgreSQL running → connects. Not running → clear error.

Step 4

Models: RFQ + Workflow + RFQ_Stage

models/*.py
Understand

Why 3 at once? Create-RFQ auto-generates stages from a workflow → needs all 3 tables. SQLAlchemy

Column types: Column(String(200)) = VARCHAR. Column(UUID, primary_key=True) = UUID PK. Column(ForeignKey("rfq.id")) = FK reference.

Alembic reads your models, compares to DB, generates migration. You never write CREATE TABLE SQL.

Build

OpenAPI spec → schemas → each field becomes a Column(). Write 3 models. Add index=True on filtered/sorted columns: rfq.status, rfq.deadline, rfq.client_name, rfq.current_stage_id, rfq_stage.rfq_id, rfq_stage.order. Indexes are like a book's table of contents — PostgreSQL jumps to matching rows instead of scanning every row. Run: alembic revision --autogenerate -m "create rfq workflow rfq_stage" → alembic upgrade head. Alembic includes the indexes automatically.

Step 8

RFQ controller (+ stage auto-creation)

controllers/rfq_controller.py
Understand

The brain. Create method is the most complex in the project:

  • Look up workflow → get stage templates
  • Create RFQ record
  • Loop stages → create rfq_stage row for each
  • Back-calculate planned_start/planned_end from deadline
  • Set current_stage_id = first stage
  • One session.commit() — everything or nothing

SQLAlchemy All session.add() share the same session. One commit. Fail → rollback → nothing saved.

Build

RfqController receives datasources + session. Methods: create (full logic), get, list, update (deadline → recalc).

Ask Claude

"Walk me through date back-calculation step by step." Understand before implementing.

Date back-calculation logic: Work backwards from the deadline using each stage's planned_duration_days. Example: deadline = day 30, stage 3 = 5 days, stage 2 = 10 days, stage 1 = 8 days → stage 3: day 25–30, stage 2: day 15–25, stage 1: day 7–15. V1 uses calendar days only — no weekends/holidays. Keep the logic in a private method _calculate_stage_dates(deadline, stages) so it's easy to unit test and extend later.

Needs minimal WorkflowDatasource.get_by_id() and RfqStageDatasource.create() — build just those methods.
